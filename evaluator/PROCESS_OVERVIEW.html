۱. چه چیزی به کاندیدا می‌دهیم؟

کاندیدا یک پکیج (zip / ریپوی public) دریافت می‌کند که شامل پروژهٔ استارتر و مستندات تسک است. پوشهٔ evaluator/ هرگز تحویل داده نمی‌شود.

فایل / پوشهتوضیح

mock-backend/بک‌اند mock بدون‌وابستگی (Node خالص) — همهٔ endpointها و دیتای seed برای اجرای لوکال.

بک‌اند به‌صورت یک mock بدون‌وابستگی داخل mock-backend/ همراه پکیج داده می‌شود (فقط node server.js). قراردادش ثابت است و کاندیدا تغییرش نمی‌دهد — فقط کلاینت را به آن وصل می‌کند. OTP در پاسخ برمی‌گردد، پس کل فلو بدون SMS و بدون دیتابیس قابل تست است.

۲. انتظار چه کاری داریم؟

تسک برای تقریباً دو روز (~۱۶ ساعت) طراحی شده. کاندیدا باید:

  • اپ را روی بک‌اند (mock همراه پکیج) اجرا کند و باگ‌ها را رفع کند تا اپ کامل کار کند.
  • UI/UX همهٔ صفحه‌های موجود را از پایه بازطراحی و بهتر کند — این بخش اصلی و سنگین‌ترین وزن نمره است. اپ استارتر عمداً بدون استایل تحویل شده.
  • یک design system خودش بسازد (tokens + کامپوننت‌های reusable).
  • یک صفحهٔ Discover جدید به‌عنوان landing بسازد (featured rail، continue-watching، دسته‌بندی، جستجو).
  • هر ۶ باگ کاشته‌شده را پیدا و رفع کند و در BUGFIX_NOTES.md مستند کند (وزن کم).
  • در صورت تمایل ساختار/معماری پروژه را هم بهتر کند (پاداش دارد).
  • حالت‌های loading / error / empty طراحی‌شده در همهٔ صفحات داشته باشد.
  • کد را روی یک ریپوی public با تاریخچهٔ کامل git پوش کند و لینکش را بدهد.
  • سه مستند README.md (با اسکرین‌شات همهٔ صفحه‌ها)، AI_USAGE.md، BUGFIX_NOTES.md را پر کند.
جمع‌بندی: این تسک UI/UX-first است. اپ از قبل کار می‌کند ولی زشت است؛ کار اصلی فرانت، بازطراحی و صیقل کل اپ است. اپِ «کار می‌کند ولی زشت» باید نمرهٔ پایین بگیرد.

۳. از فرانت دقیقاً چی می‌خوایم؟

اپ استارتر همهٔ صفحات و لایهٔ داده را از قبل دارد و کار می‌کند، ولی عمداً بدون استایل است (ویوهای خام React Native). این صفحه‌ها بِیس هستند؛ تعریف می‌کنند اپ چه‌کار می‌کند، نه این‌که چقدر خوب به‌نظر می‌رسد. کار اصلی فرانت اینه که اینها را به یک محصول واقعی و صیقلی تبدیل کند.

الف) بازطراحی UI/UX همهٔ صفحه‌ها (بخش اصلی — ~۶۰ امتیاز)

همهٔ این صفحه‌ها باید از حالت خام به طراحی production تبدیل شوند: Phone، OTP، Catalog، Game Detail، Video Player، Favorites، Wishlist، Watch History، Profile، و Discover.

  • طراحی بصری در همهٔ صفحه‌ها (۳۵): layout، ریتم spacing، مقیاس تایپوگرافی، رنگ/کنتراست، سلسله‌مراتب، آیکن، و انسجام بصری بین همهٔ صفحه‌ها. RTL فارسی درست.
  • Design system و کامپوننت‌های reusable (۱۵): tokens متمرکز (رنگ/spacing/تایپو/شعاع/سایه) + کتابخانهٔ کوچک کامپوننت (Button، Card، Input، Badge، …). بدون magic number پراکنده یا کپی‌پیست استایل.
  • States و interaction/motion (۱۰): loading/error/empty طراحی‌شده (skeleton/shimmer)، حالت pressed/disabled، keyboard avoidance، pull-to-refresh، optimistic UI.

ب) صفحهٔ Discover جدید (۱۵)

یک landing جدید: featured rail (GET /games/featured)، continue-watching (GET /watch-history، خالی شد مخفی)، دسته‌بندی از GET /games با categoryConfig، و جستجوی debounce‌شده. تب اولیهٔ authenticated باشد و از API client و state مشترک استفاده کند.

ج) موارد ثانویه (وزن کم)

  • رفع ۶ باگ (۵): برای این‌که اپ واقعاً کار کند؛ مستند در BUGFIX_NOTES.md.
  • معماری و ساختار (۵): حفظ لایه‌بندی و typecheck تمیز. اگر ساختار/فولدربندی را بهتر کند (مثلاً feature-based) و توجیه کند، پاداش می‌گیرد — می‌خواهیم ببینیم کی ساختار بهتری انتخاب می‌کند.
  • Git workflow، AI usage، و README با اسکرین‌شات همهٔ صفحه‌ها (هر کدام ۵).
قانون gating: اگر اپ هنوز شبیه استارترِ بدون‌استایل باشد، دسته‌های ۱ تا ۴ نمی‌توانند از ~۳۰٪ بالاتر بروند — هرچقدر هم کد تمیز باشد. هدف این تسک، صنعتگریِ فرانت است.
داده‌های mock غنی است (۸ بازی، ۴ دسته، ۴ featured، تاریخچهٔ seed‌شده). کاندیدا می‌تواند کتابخانهٔ UI/آیکن/انیمیشن اضافه کند اگر در README توجیه کند، ولی طراحی (layout و زبان بصری) باید کار خودش باشد.

۴. باگ‌های کاشته‌شده (کلید پاسخ — محرمانه)

۶ باگ در حوزه‌های auth، API، state و UI پخش شده‌اند. هر باگ به یک property test متصل است که با رفع باگ سبز می‌شود.

باگحوزهعلامتریشه (خلاصه)
BUG-01 Auth بعد از restart همیشه logout می‌شود. توکن‌ها فقط in-memory نگه داشته می‌شوند؛ persist/hydrate به secure-store انجام نمی‌شود.
BUG-02 API صفحات محافظت‌شده همیشه 401 می‌دهند. هدر بدون فاصله ساخته می‌شود: "Bearer"+token. باید Bearer <token> با یک فاصله باشد.
BUG-03 API بعد از ~۱۵ دقیقه به‌جای refresh، logout می‌شود. توکن جدید روی retry اعمال نمی‌شود، یا پاسخ refresh (فقط accessToken) اشتباهاً ناقص فرض و session پاک می‌شود.
BUG-04 State favorite برمی‌گردد یا بازی اشتباه favorite می‌شود. به‌روزرسانی optimistic با index کلید می‌خورد نه gameId، و روی خطا revert نمی‌شود.
BUG-05 UI عنوان/دستهٔ بازی خالی رندر می‌شود. mapper از فیلدهای enum قدیمی category/difficulty می‌خواند؛ باید از categoryConfig/difficultyConfig بخواند.
BUG-06 UI اسپینر کاتالوگ هیچ‌وقت محو نمی‌شود / روی نتیجهٔ خالی crash می‌کند. نبودِ finally برای پاک‌کردن loading + map روی دادهٔ undefined بدون گارد.
جزئیات کامل ریشه، مسیر فایل و فیکس مینیمال هر باگ در evaluator/BUG_ANSWER_KEY.md است. این جدول فقط خلاصه است.

نکات کلیدی نمره‌دهی

  • UI-first: ~۶۰ از ۱۰۰ امتیاز کیفیت بصری و تعاملی است. اپِ «کار می‌کند ولی زشت» باید نمرهٔ پایین بگیرد.
  • قانون gating: اگر اپ هنوز شبیه استارترِ بدون‌استایل باشد، دسته‌های ۱ تا ۴ از ~۳۰٪ بالاتر نمی‌روند.
  • چک‌اوت تمیز: روی یک clone تازه و طبق README خودِ کاندیدا نمره می‌دهیم؛ هیچ فایلی دستی اصلاح نمی‌کنیم.
  • مبتنی بر شواهد: برای هر کسر امتیاز، چیزی که در اپ دیده شده (صفحه، اسکرین‌شات، فایل/خط) ثبت می‌شود.
  • Stretch goals: جدا یادداشت می‌شوند و هرگز مجموع را بالای ۱۰۰ نمی‌برند.

۶. پروسهٔ عملی گرفتن نمره

  1. mock backend را بالا بیاور (cd mock-backend && node server.js) و مطمئن شو POST /auth/send-otp مقدار otp را برمی‌گرداند.
  2. از سابمیشن کاندیدا یک clone تازه بگیر؛ دستور install مستندشده را اجرا کن (بدون اصلاح دستی).
  3. اپ را روی بک‌اند اجرا کن و typecheck را بگیر.
  4. هر صفحه را در اپِ در حال اجرا نگاه کن و کیفیت بصری/تعاملی‌اش را با باندهای رابریک تطبیق بده — این بخش اصلی قضاوت است. اسکرین‌شات‌های README کمک می‌کنند ولی در اپ تأیید کن.
  5. هر باگ را با BUG_ANSWER_KEY.md راستی‌آزمایی کن (آیا علامت رفع شده؟).
  6. هر دسته را مستقل نمره بده و در GRADING_WORKSHEET.md با شواهد ثبت کن.
  7. مجموع را از ۱۰۰ حساب کن؛ stretch goals را جداگانه یادداشت کن.
برای نمره‌دهی، نسخهٔ خصوصی evaluator/ خودت را استفاده کن. سابمیشن کاندیدا هیچ فایل evaluator ندارد، پس چیزی برای merge برگشتی وجود ندارد.

۷. تحویل امن (یادآوری)

قبل از فرستادن هر چیزی به کاندیدا، مطمئن شو پوشهٔ evaluator/ داخلش نیست.

# پوشه نباید در working tree باشد
test ! -e evaluator/ && echo "OK: absent" || echo "FAIL: present"

# git نباید روی برنچ تحویل آن را track کند
git ls-files | grep -q '^evaluator/' && echo "FAIL: tracked" || echo "OK"

# هیچ ارجاع به کلید پاسخ/رابریک نباشد
grep -rli --include='*.md' 'BUG_ANSWER_KEY\|GRADING_WORKSHEET' . && echo "FAIL" || echo "OK"
