۱ چه چیزی به کاندیدا می‌دهیم؟

۲ انتظار چه کاری داریم؟

تسک برای تقریباً دو روز (~۱۶ ساعت) طراحی شده. کاندیدا باید:

  • باگ‌ها را زود رفع کند تا اپ کامل و قابل‌استفاده شود.
  • UI/UX همهٔ صفحه‌های موجود را از پایه بازطراحی و بهتر کند — بخش اصلی و سنگین‌ترین وزن نمره.
  • یک design system خودش بسازد (tokens + کامپوننت‌های reusable).
  • صفحهٔ Discover را به‌عنوان landing بسازد (featured rail، continue-watching، دسته‌بندی، جستجو).
  • حالت‌های loading / error / empty طراحی‌شده در همهٔ صفحات داشته باشد.
  • در صورت تمایل ساختار/معماری پروژه را هم بهتر کند (پاداش دارد).
  • کد را روی یک ریپوی public با تاریخچهٔ کامل git پوش کند و لینک بدهد.
  • سه مستند را پر کند: README.md (با اسکرین‌شات همهٔ صفحه‌ها)، AI_USAGE.md، BUGFIX_NOTES.md.
قانون طلایی: این تسک UI/UX-first است. اپِ «کار می‌کند ولی زشت» باید نمرهٔ پایین بگیرد — کارِ حداقلی (فقط فیکس باگ) نباید نمرهٔ بالا بگیرد.

۳ از فرانت دقیقاً چی می‌خوایم؟

اپ استارتر همهٔ صفحات و لایهٔ داده را دارد و کار می‌کند، ولی با ویوهای خام React Native و بدون طراحی. این صفحه‌ها بِیس هستند — تعریف می‌کنند اپ چه‌کار می‌کند، نه اینکه چقدر خوب دیده می‌شود. کار فرانت، تبدیل این بِیس به یک محصول واقعی است.

الف) بازطراحی UI/UX همهٔ صفحه‌ها بخش اصلی · ~۶۰ امتیاز

این ۱۰ صفحه باید از حالت خام به طراحی production برسند:

Phone OTP Catalog Game Detail Video Player Favorites Wishlist Watch History Profile Discover

  • طراحی بصری (۳۵): layout، ریتم spacing، مقیاس تایپوگرافی، رنگ/کنتراست، سلسله‌مراتب، آیکن، و انسجام بصری بین همهٔ صفحه‌ها. RTL فارسی درست.
  • Design system (۱۵): tokens متمرکز (رنگ/spacing/تایپو/شعاع/سایه) + کتابخانهٔ کوچک کامپوننت (Button، Card، Input، Badge، …). بدون magic number پراکنده یا کپی‌پیست استایل.
  • States و interaction/motion (۱۰): loading/error/empty طراحی‌شده (skeleton/shimmer)، حالت pressed/disabled، keyboard avoidance، pull-to-refresh، optimistic UI.

ب) صفحهٔ Discover جدید ۱۵ امتیاز

یک landing جدید: featured rail (GET /games/featured)، continue-watching (GET /watch-history — خالی شد مخفی)، دسته‌بندی از GET /games با categoryConfig، و جستجوی debounce‌شده. تب اولیهٔ authenticated باشد و از API client و state مشترک استفاده کند.

ج) موارد ثانویه وزن کم

  • رفع ۶ باگ (۵): برای اینکه اپ واقعاً کار کند؛ مستند در BUGFIX_NOTES.md با کامیت‌های fix: مجزا.
  • معماری، صحت و performance (۵): حفظ لایه‌بندی و typecheck تمیز + بهینگی کل اپ (لیست‌های virtualized، memoization، بدون re-render اضافی، بدون fetch تکراری). اگر ساختار/فولدربندی را بهتر کند (مثلاً feature-based) و توجیه کند، پاداش می‌گیرد — می‌خواهیم ببینیم کی ساختار بهتری انتخاب می‌کند.
  • Git workflow · AI usage · README با اسکرین‌شات (هر کدام ۵).
قانون gating: اگر اپ هنوز شبیه استارترِ بدون‌استایل باشد، دسته‌های ۱ تا ۴ نمی‌توانند از ~۳۰٪ بالاتر بروند — هرچقدر هم کد تمیز باشد.
دیتای mock غنی است (۸ بازی، ۴ دسته، ۴ featured، تاریخچهٔ seed‌شده) تا صفحه‌ها پر و قابل‌قضاوت باشند. کاندیدا می‌تواند کتابخانهٔ UI/آیکن/انیمیشن اضافه کند اگر در README توجیه کند، ولی زبان بصری و layout باید کار خودش باشد.

۴ باگ‌های کاشته‌شده کلید پاسخ — محرمانه

۶ باگ در حوزه‌های auth، API، state و UI پخش شده‌اند. این‌ها صرفاً برای این هستند که اپ درست کار کند تا کاندیدا بتواند رویش UI بسازد؛ وزن نمره‌شان کم است (۵).

باگحوزهعلامتریشه (خلاصه)
BUG-01 Auth بعد از restart همیشه logout می‌شود. توکن‌ها فقط in-memory نگه داشته می‌شوند؛ persist/hydrate به secure-store انجام نمی‌شود.
BUG-02 API صفحات محافظت‌شده همیشه 401 می‌دهند. هدر بدون فاصله ساخته می‌شود: "Bearer"+token. باید Bearer <token> با یک فاصله باشد.
BUG-03 API بعد از ~۱۵ دقیقه به‌جای refresh، logout می‌شود. توکن جدید روی retry اعمال نمی‌شود، یا پاسخ refresh (فقط accessToken) اشتباهاً ناقص فرض و session پاک می‌شود.
BUG-04 State favorite برمی‌گردد یا بازی اشتباه favorite می‌شود. به‌روزرسانی optimistic با index کلید می‌خورد نه gameId، و روی خطا revert نمی‌شود.
BUG-05 UI عنوان/دستهٔ بازی خالی رندر می‌شود. mapper از فیلدهای enum قدیمی category/difficulty می‌خواند؛ باید از categoryConfig/difficultyConfig بخواند.
BUG-06 UI اسپینر کاتالوگ هیچ‌وقت محو نمی‌شود / روی نتیجهٔ خالی crash می‌کند. نبودِ finally برای پاک‌کردن loading + map روی دادهٔ undefined بدون گارد.
جزئیات کامل ریشه، مسیر فایل و فیکس مینیمال هر باگ در evaluator/BUG_ANSWER_KEY.md است. این جدول فقط خلاصه است.

نکات کلیدی

  • UI-first: ~۶۰ از ۱۰۰ امتیاز کیفیت بصری/تعاملی است.
  • قانون gating: اگر اپ هنوز شبیه استارترِ بدون‌استایل باشد، دسته‌های ۱ تا ۴ از ~۳۰٪ بالاتر نمی‌روند.
  • چک‌اوت تمیز: روی یک clone تازه و طبق README خودِ کاندیدا نمره می‌دهیم؛ هیچ فایلی دستی اصلاح نمی‌کنیم.
  • مبتنی بر شواهد: برای هر کسر امتیاز، چیزی که در اپ دیده شده (صفحه، اسکرین‌شات، فایل/خط) ثبت می‌شود.
  • Stretch goals: جدا یادداشت می‌شوند و هرگز مجموع را بالای ۱۰۰ نمی‌برند.

۶ پروسهٔ عملی گرفتن نمره

  1. mock backend را بالا بیاور (cd mock-backend && node server.js) و مطمئن شو POST /auth/send-otp مقدار otp را برمی‌گرداند.
  2. از سابمیشن کاندیدا یک clone تازه بگیر؛ دستور install مستندشده را اجرا کن (بدون اصلاح دستی). اگر طبق README اجرا نشد، خودش یک سیگنال نمره است.
  3. اپ را اجرا کن و npm run typecheck را بگیر.
  4. هر صفحه را در اپِ در حال اجرا نگاه کن و کیفیت بصری/تعاملی‌اش را با باندهای رابریک تطبیق بده — این مرحلهٔ اصلی قضاوت است. اسکرین‌شات‌های README کمک می‌کنند ولی در اپ تأیید کن.
  5. هر باگ را با BUG_ANSWER_KEY.md راستی‌آزمایی کن (آیا علامت دیگر رخ نمی‌دهد؟).
  6. صفحهٔ Discover و design system را جداگانه بررسی کن (انسجام tokens، reuse کامپوننت‌ها).
  7. هر دسته را مستقل نمره بده و در GRADING_WORKSHEET.md با شواهد ثبت کن؛ مجموع از ۱۰۰.
برای نمره‌دهی، نسخهٔ خصوصی evaluator/ خودت را استفاده کن. سابمیشن کاندیدا هیچ فایل evaluator ندارد.
